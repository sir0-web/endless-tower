## 1. コンセプト（確定・北極星）

> **「一人では攻略できない塔を、仲間の"意志"を受け継ぎながら全員で踏破する非同期協力型ローグライク」**

### 狙い

- プレイヤーはそれぞれ**自分のタイミング**で塔へ挑戦する
- **敗北は失敗ではなく、次の挑戦者へ力を託す行為**となる
- 誰かの犠牲が、誰かの前進になり、最終的には**ギルド全員の力で塔を攻略**する

### キーワード

| | | |
|---|---|---|
| 非同期協力 | 死亡＝貢献 | 継承 |
| ローグライク | コミュニティ攻略 | リレー形式 |

**「次へ繋ぐ」ゲーム体験**

### プレイヤー体験の転換

| 従来のゲーム | KIZUNA（新章） |
|---|---|
| 戦う → 強くなる → **自分でクリアする** | 力尽きても次の仲間へ力を託す → 受け取った力でさらに前へ → みんなの意志を繋いで頂上へ → **ギルド全員の力で塔を攻略** |
| 自分が強くなってクリアする | **自分が次の挑戦者を強くして、みんなでクリアする** |

### 体験イメージ（コンセプト画像より）

```
「15Fで力尽きた…でも、宝と経験を残しておくよ！」（10分前）
　↓
「ありがとう！その力で16Fを突破したよ！」（30分前）
　↓
「おかげで20Fまで来れた！次はきっと頂上へ…！」（1時間前）

—— みんなの意志が、塔を登っていく。
```

## 2. 新章の位置づけ（確定）

- **EbT本編とは別ゲームではなく、EbT内の1コンテンツ**として制作する
  - 入口イメージ：EbTのTOPからモード選択、またはひろばから**「オンラインの世界へ」（仮）**を選んで新章（共闘・協力）へ**転生**する仕組み
- ゲーム型式は**協力クリア型ローグライク＝やり切り型**（クリアが存在する。アップデートで上層階を追加する余地あり）
  - ※エンドレス型の本編とはここが根本的に異なる
- **ストーリーモードあり**：コンテンツ名称で本編と切り分ければ違和感なし。「普段遊んでいるコンテンツの厚みが出る」（まっち）
  - 理想の体験アーク：**ストーリー性 → クリアへの盛り上がり → エンドロールの達成感**（モコ）
- **マップ・基礎構成は現行EbTの仕様にまるまる乗せる**（新規マップシステムの開発はしない）

## 3. 継承システム（方向性確定：③＋④の組み合わせが核）

モコ調査による継承4類型と、コンセプト（§1）を軸にした採否判断。

| 型 | 内容 | 採否 | 理由・方針 |
|---|---|---|---|
| ① ステータス合算型 | 死んだ仲間のステータスの一部が次の仲間のベース能力に永久加算 | △ 主軸にしない | 個人内で完結しがちで「他者への貢献」を表現しにくい。**ギルドツリー解禁後の微増ボーナス扱い**に留める |
| ② スキル選択・融合型 | 死者の必殺技/パッシブを遺品・魂としてリスト化し選んでセット | △ 初期は見送り | 物語性は最強クラスだがマッチング・演出コストが重く、非同期協力と絡めると複雑化しすぎる。将来的に**「形見に付与するパッシブ1つ」程度の軽量版**として③④に統合 |
| ③ ツリー解禁型（家系図） | 死者の生前の実績に応じたポイントで一族全体の強化ツリーをアンロック | **◎ 採用** | 「一族＝ギルド全体」と読み替えることで"全員の力で塔を攻略する"を体現。**誰の死も共有ツリーに貢献＝死亡＝貢献が自然に成立** |
| ④ リソース限定型（形見セレクト） | 死者の装備1つ or 最高ステータス1つだけを選択して継承 | **◎ 採用** | **ドッペルゲンガーと直結**しやすい。撃破時に"託されたアイテム"を選ばせれば「誰かの犠牲→誰かの前進」を1アクションで体験できる |

### 3-1. ③ギルド共有「塔攻略ツリー」

- 誰が死んでも**貢献値がギルド全体に加算 → ツリー解禁**
- 「みんなで攻略する」の骨格
- ツリー内容の候補：固定の全員共有ステータス or パッシブスキル（モコ案）
- **転生後のフルポイント分岐**で個人のプレイスタイルの幅も担保できる（モコ案・要詳細化）

### 3-2. ④形見セレクト（ドッペルゲンガー拡張）

- 既存のドッペルゲンガー（死亡キャラ登録→他PLの±10階に出現→撃破）の上に載せる
- 現状は「撃破ボーナス（固定額付与）」に近いので、**撃破時にモーダルを挟んで以下から1つ選択**する形へ拡張：
  1. **装備1つ**（equipmentから選択）
  2. **最高ステータス1点**（str/agi/dex/int/vit/luk の最大値）
  3. **stat_point_reward の一部 or 全部**
- equipmentは既にDB保存されているため、形見セレクトUIを乗せる追加コストは小さい
- モコ：「まさに理想」（確定）

## 4. 魂のメッセージ（確定＋詳細方針）

### 基本仕様

- **死亡時、その場（死亡地点）にメッセージを残せる**
- 用途：次に来る人への応援、「この先のXXXやべーぞ！」的な警告・攻略情報
- ボス前や特殊モンスターで行き詰まった際のコメント用途も想定
- デバッグ環境での画面イメージは作成済み（まっち、2026-07-12朝）

### 増殖・容量対策（方針決定）

- DB保存上限を考慮し、文字数とエフェクト整理後に**新規50体くらい保存**できるよう調整
- 削除は**時限式（48時間 or 72時間を想定）**が省エネで基本方針
  - 「他の冒険者が触れた回数で削除」も実装可能だが、現状の接続者数では触れられず残り続ける可能性があるため不採用

### 発展アイデア（案として保持・要設計）

- **魂のログ（記録書）**：誰かが触れたメッセージだけが全員共有の書物に記録され、いつでもテキストで閲覧できる。「みんなで発見したものをシェアしあう」体験
  - ⚠️ 「見つける意義」と「埋めてシェアするメリット」を研ぎ澄まさないと他人任せコンテンツになるため設計必須
- **ゲーム外での閲覧**：ランキングのようにセットアップ画面から死亡時ログを振り返れる「思い出」機能（モコ案）
- **クリア後エンドロール**：スタッフロールのように死亡ログとメッセージが流れる演出（モコ案）。実装自体は容易（まっち確認済み）。§2の「エンドロールの達成感」と直結する有力候補

## 5. 拠点成長システム（箱庭）— 新章の追加要素として提案・評価済み

スタート地点を「ふもとの村/町」に設定、またはスタート階層に施設を設置し、**共有スペースの協力育成**で「一緒にやっている感覚」を強化する（モコ提案 2026-07-12）。

**施設候補（仮）**
- 強化施設：武器屋・防具屋・鍛冶屋・雑貨屋・精錬など
- 資源施設：畑・鉱山・森など

**解放条件（仮）と評価**

| 案 | 評価 | 備考 |
|---|---|---|
| ① 進行度による解放 | ◎ | 王道 |
| ② プレイ回数による解放 | ✕ | 自殺周回で破綻（課題3） |
| ③ ダンジョン内で発見して解放 | ◎ 最推奨 | 「捕まっている大工を助けて村へ送る」等。探索動機と直結 |
| ④ 資源投入で解放 | ◎ | ③とセットで機能 |

**成長システム（仮）と評価**

| 案 | 評価 | 備考 |
|---|---|---|
| ① プレイ回数 | ✕ | 同上 |
| ② 死亡時に進行度ポイント付与→投入で成長 | ◎ 最推奨 | 深いほど貢献大＝**「死亡＝貢献」と完全一致**。③ギルドツリーの貢献値と統合できる可能性 |
| ③ 資源を一定投入で成長 | ○ | 資源システムの新規実装が前提 |
| ④ 時間による成長（プレイ回数で短縮） | △ | 救済用途（オフライン中の緩回復程度）に限定なら可 |

**導入方針（まっち所感）**
- 施策の想い（協力・共闘の深化）と現状のプレイログ（単独プレイ時間が多い）が合致していないため、**ユーザーが時間を合わせたくなるプロモーションも併せて必要**
- **フェーズ1として「捕まっている大工を助けて村へ送る」型の個人の町拡張をまず導入して様子見**が有力

## 6. 設計上の要注意ポイント（実装前に解決すべき課題）

1. **ランキングの公平性（最重要）**：継承・拠点の恩恵でスタート時の強さが変わると「同じB50F到達」の価値が変わり歴代記録と比較不能になる
   - 対策案a：恩恵を「利便性」（倉庫・初期持ち込み選択・序盤の精錬アクセス等）に限定し、生のステータス強化は入れない
   - 対策案b：ランキングに村Lv併記、またはシーズン制で世代分離（歴代をviewからコピーした運用前例あり）
   - ※新章は「やり切り型」の別コンテンツ（§2）なので、**新章専用の進行度表示にすれば本編ランキングとの衝突は回避可能**
2. **クライアント信頼問題**：ゲームロジックは全てクライアント側でチート耐性がない。共有進捗（ギルドツリー・村）への貢献は**サーバー側RPCで上限・レート制御が必須**（例：1プレイヤー1日あたり投入上限、進行度に応じた投入可能量。ジャックポット冪等化と同型の設計で対応可能）
3. **自殺周回対策**：貢献量は「回数」ではなく**「深度」を通貨**にする（浅い死は貢献が小さい設計）
4. **イベント競合**：すかるぽりん等の既存全鯖イベントとの同時発生時の挙動整理（過去に「逃走先ユーザーへの移動ロジック」を湧き時間20分固定の制約＋イベント競合バグ配慮で断念した経緯あり）

## 7. 進め方・体制（確定）

- **順番が大切**：出たアイデアから**根幹となるメインコンテンツをフォーカス**し、優先実装 → 関連コンテンツを芋づる式で順に実装
- そのための**相関図**を作成中（Claude整理）。完成し「イイ感じ」になったタイミングで、関係性の強いところから順に実装
- **スケジュール感**：弓実装 → しばらく細かい修正 → **来週（7月中旬〜）くらいから新機能着手**が目標（2026-07-12時点）
- **作画**：ゲームに必要な作画数は膨大だが、base数点＋生成AIで負担を抑えて実装する方針
- **役割分担**：
  - まっち：実装・設計・実現性判断・企画書への集約
  - モコ：現実性を考慮しないアイデア・ユーザー視点の理想形の提供（システム/ストーリー/職業/アイテム/グラフィックなんでも）。ツリースキルのアイデア等を思考中
